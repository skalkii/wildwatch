# GenAI Roadmap — beyond indexing + search

> Status: **deferred to next phase.** Pipeline + alerts + dashboard are
> the priority right now. This file captures every VideoDB GenAI capability
> we have **not yet wired**, with a concrete WildWatch use case for each and
> a rough effort estimate so we can rank them when the current build is bug-
> free and demo-ready.

VideoDB's SDK exposes three GenAI surfaces beyond the perception primitives
(`index_scenes` / `index_audio` / `search`) that WildWatch already uses:

| Method (on `coll`) | Purpose | WildWatch use |
| --- | --- | --- |
| `generate_text(prompt, model_name, response_type)` | LLM completion | Already wired ✓ — alert rewriter. More uses below. |
| `generate_voice(text, voice_id, ...)` | Text-to-speech | Daily-digest voiceover narration, alert TTS for blind/low-vision rangers. |
| `generate_image(prompt, ...)` | Text-to-image | Cover-card image for daily digests; share-on-Twitter alert preview. |
| `dub_video(video_id, language_code)` | Multilingual dub of an uploaded video | Translate daily reels for multi-language ranger crews. |
| `generate_music(prompt, duration, ...)` | Background score | Digest reel soundtrack (already wired in `wildwatch/digest.py` — verify when phase starts). |
| `generate_sound_effect(prompt, duration, ...)` | One-shot SFX | Alert-tier audio stings for the daily reel (`[urgent]` → siren-like cue). |
| `generate_video(prompt, ...)` | Text-to-video | "What if" demo clip generation — show the difference between a normal vs threatened scene to a stakeholder during a pitch. |

---

## Phase 1 — Done ✅

**Alert text rewriter via `generate_text`.**

- File: `wildwatch/telegram.py:_genai_friendly_explanation`.
- Trigger: every `send_alert` call whose `explanation` contains a `[` (i.e.
  Path-B sweep output with bracket-tagged AI text). RTStream callbacks
  bypass since VideoDB's own explanation is already short.
- Model: `basic` (cheap, fast). Falls back to local parser on timeout/error.
- Telegram link preview disabled so the console-player URL no longer
  renders an empty box.

## Phase 2 — Daily highlight digest with voiceover

Goal: at 23:55 every day, the system auto-produces a 60–120 s reel that
contains:

1. **Sorted clips** from the day's events (top tier first, then by
   confidence). Source: the existing `data/live_event_log.jsonl`.
2. **Voiceover narration** generated by `coll.generate_voice(...)` from a
   `generate_text`-produced script. The script summarises:
   - How many alerts fired in each tier.
   - The most-spotted species across the day.
   - Any anomalies (camera_health_issue, water_critical, etc.).
3. **Background music** (`coll.generate_music`) — already wired in
   `wildwatch/digest.py` behind `--music`. Verify before phase starts.
4. **Cover image** (`coll.generate_image`) — a single hero frame the
   Telegram message can preview (Instagram-style). Prompt: "ecology
   illustration of <top-species> at <site_context>, golden hour, painted
   poster style". Phase 2.5 polish.

Trigger paths:
- Cron via `scripts/build_digest.py --since-hours 24` (existing).
- Manual button on the dashboard's Alerts tab ("Build today's reel now").
  Routes to `POST /api/digest/build` which spawns the existing builder
  in a background task.

New endpoints / files:
- `POST /api/digest/build` (sync return = task id + ETA, async finish).
- `GET /api/digest/{id}` for poll status.
- Telegram delivery: send the player_url + cover image as a single
  message at end of day.

Effort: ~6h.

## Phase 3 — Per-event category playback compilation

Goal: any alert label can be expanded into "all hits this week" — e.g.
tap `predator_activity` and get a 30 s compilation of every predator-
activity scene from the last 7 days.

Implementation:
1. `GET /api/playlist/{label}?since_hours=168` — filters event_log by
   label, builds a VideoDB programmable-editor `Timeline` from each hit's
   `(video_id, start, end)`, returns the compiled `stream_url`.
2. Dashboard: each event card on the Alerts tab gets a "watch all"
   icon → opens the modal HLS player with the compiled stream.
3. Optional voiceover via `generate_voice("Today's predator activity:
   three sightings — leopard at the Etosha waterhole at 6 a.m., ...")`.

Effort: ~4h (most of the Timeline plumbing already exists in
`wildwatch/digest.py`).

## Phase 4 — Multilingual delivery via `dub_video`

Goal: rangers in Spanish-speaking, Swahili-speaking, Hindi-speaking
reserves get the daily reel auto-dubbed.

Implementation:
1. After Phase 2's digest finishes, fan out `coll.dub_video(video_id,
   language_code='es')` (and other codes from a per-recipient
   config).
2. Telegram bot maps each chat_id to its preferred language_code in
   `.env` (`TELEGRAM_CHAT_ID_EN`, `TELEGRAM_CHAT_ID_ES`, …) or in
   `.state.json["subscribers"]`.
3. Each chat receives the dub in its own language.

Effort: ~3h once Phase 2 is shipped.

---

## Phase 5 — Generative demo content via `generate_video`

Goal: pitch / fundraising / training material. "Show me what a
mortality event looks like even though we haven't captured one
organically yet."

Use sparingly — generated footage is NOT for operational alerts. Phase
5 is purely for stakeholder communication.

Implementation:
- One-off script `scripts/generate_demo_clip.py --prompt "<...>"
  --duration 8 --label mortality_event`.
- Outputs a video_id we can pipe into the digest builder so the demo reel
  contains a synthetic mortality example clearly labelled "ILLUSTRATIVE".

Effort: ~2h. Strictly opt-in via flag.

---

## Phase 6 — Voice-driven alert acknowledgement (stretch)

Goal: ranger replies to a Telegram alert with a voice note ("acknowledged,
moving in"); we transcribe it via `generate_text` (or `index_spoken_words`
on the recording) and append to the event log for auditability.

Effort: ~5h. Only useful at scale.

## Out-of-scope reminders

- Stay inside VideoDB SDK — don't add ElevenLabs / OpenAI / Replicate as
  secondary providers, even when their model is technically better.
  Single-vendor footprint is part of the hackathon depth-of-usage axis.
- Cache GenAI outputs aggressively. Same alert label + same scene
  description = same rewrite; reuse via a small dict keyed on the
  hash of `(label, scene_summary)`. Wire this in Phase 2.
- For every GenAI surface added, log the per-call $ estimate to the Usage
  tab so the operator sees the cumulative GenAI spend alongside the
  existing perception spend.
